PostgreSQL 멀티테넌시 구성 예시 (ft. SaaS 구성)

2023년 11월 13일 · 약 3분

멀티테넌시의 의미를 알아보고 PostgreSQL 에서 멀티테넌시를 구성하는 방법에 대해서 알아봅니다.

멀티테넌시(Multi-Tenancy)?​

먼저, 테넌시(Tenancy)란 '차용, 소작'을 의미합니다. '임차'라는 단어를 사용하면 조금 더 이해가 쉬울 것 같습니다. 여러 회사 또는 개인이 건물에 각각의 호실을 임차하는 것을 '멀티테넌시(Multi-Tenancy)'로 비유할 수 있을 것 같습니다.

멀티테넌시 구성 전략​

멀티테넌시를 구성하는 전략은 대표적으로 Row Level, Schema Level, Database Level 3가지가 있습니다.

Row Level​

단일테넌시에서 Foreign Key를 사용해서 데이터 관계을 구성하듯이, Tenant를 관리할 별도의 테이블을 작성하고, tenant_id를 각 데이터 행에 부여함으로써 Row의 소유권을 가지게됩니다.

Schema Level​

PostgreSQL의 경우, Database 와 Table 사이에 Schema 라는 일종의 네임스페이스를 구성할 수 있습니다. 이를 통해 단일 DB 내에서 Schema를 통해 Tenant를 구성할 수 있습니다.

Database Level​

모든 테넌트에 대해 완전히 다른 DB를 설정합니다.

필자가 선호하는 전략​

인적 자원의 Role의 구분이 뚜렷한 (DBA) + SQL / (BE) DTO + DAO + Controller / (API) API 가 관리 측면에서 중요할 수도 있습니다. 하지만, 서비스의 확장성, 다양성을 위해 개발의 속도가 중요하다고 판단 시, 아래와 같은 구성을 선호하는 편입니다.

ORM 을 통해 Database의 기본적인 Table Schema를 관리하고, 필요 시 Schema(네임스페이스) 단위로 복제하는 형식을 선호합니다.

다만 ORM 를 통해 DB를 관리하는 경우, Transaction, Indexing 등 가장 기본적인 부분을 놓칠 수 있어 조심해야합니다.
